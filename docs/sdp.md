# SatSim Software Development Plan (SDP)

- Configuration item: SATSIM-SDP, Issue 1 (draft)
- Scope: the **SatSim satellite simulator** (ground support software). The future
  on-board software (OBSW) is explicitly **out of scope** and will have its own
  SDP with its own criticality classification and stricter tailoring.

## 1. Purpose, Product, Criticality

SatSim is a satellite simulator supporting development and automated testing of
satellite on-board software, with a PUS-C (ECSS-E-ST-70-41C) TM/TC interface, a
thin web frontend, and pluggable OBSW targets ('execution back-ends' in the accepted ADRs: in-process loopback,
native process, instruction-level emulators: TSIM, TEMU/cOBC, QEMU).

Criticality: **Category D** per ECSS-Q-ST-80C (software whose failure has
negligible safety/mission consequences — ground development tool). Rationale:
SatSim produces no flight items and its outputs are always verified by
downstream OBSW V&V.

## 2. Applicable Standards and Tailoring

Applicable: ECSS-E-ST-40C (SW engineering), ECSS-Q-ST-80C (SW product
assurance), ECSS-E-ST-70-41C (PUS, interface standard, not tailored — see
ADR-0002), CCSDS 133.0-B, CCSDS 301.0-B.

### 2.3 Change control

Changes to baselined scope or content of controlled documents (this SDP, SRS,
SVS, ICD) go through a **Software Change Request**: an SCR file in
`docs/scr/` (registered in `docs/scr/SCR-LOG.md`) records the change,
rationale, and per-CI impact analysis. Disposition is given by the project
lead via PR review; implementing document updates reference the SCR ID.
Defects/nonconformances remain with the issue tracker (§2.1); ADRs remain
governed by their own supersession rule.

## 3. Organization, Tools, Conventions

- Roles: project lead (all ECSS roles: customer, supplier, PA). AI assistant
  (Claude / Claude Code) as development tool under §6 controls.
- Language/platform: Java 21 LTS. Build: **Maven** multi-module.
- Modules: `pus-core` (packet library, framework-free), `simulator` (Spring
  Boot backend, scheduler, links; serves the static web frontend from
  `src/main/resources/static` — the "web-frontend" component of ADR-0001 is
  realized as static resources in the PoC), `sim-test-support` (traceability
  annotations).
- Coding standard: Google Java Style; static analysis: Checkstyle + SpotBugs in
  CI; custom rule: no wall-clock access in simulation logic (ADR-0006 C1).
- CM: trunk-based with short-lived branches; merge via PR; baseline = annotated
  Git tag `M<n>` incl. generated test report and traceability matrix.

## 5. Verification & Validation Plan

- Two levels: **unit tests** (untraced, engineering hygiene; coverage report) and
  **validation tests** (traced; one+ per SRS requirement; defined spec-first in
  `docs/svs.md`).
- Annotations: `@Requirement("SIM-REQ-…")`, `@TestCase("SIM-TC-…")` from
  `sim-test-support`.
- CI consistency gate: every SVS test case has an implementing test and vice
  versa; every SRS requirement is referenced by ≥1 SVS case (else build warning,
  failure at milestone gate).
- Test report: generated from Surefire results per baseline into
  `docs/test-reports/M<n>-report.md` (date, commit, environment, per-case verdict,
  coverage). Manual cases (frontend smoke test) recorded in the same report.
- Traceability matrix: generated (requirement → test case → test method →
  verdict); committed per baseline.
- Determinism policy: identical inputs ⇒ identical TM byte streams and
  timestamps (ADR-0006 C6); dedicated replay test category.

## 6. AI-Assisted Development Policy

AI (Claude / Claude Code) is used as a development tool. Controls:

1. All AI-generated code and documents enter the baseline only via PR with
   human review.
2. `docs/icd.md` §6/§7 vectors and `docs/svs.md` expected results are
   human-approved reference data; the AI must never modify them to make tests
   pass. Discrepancies are raised as findings.
3. Requirement/test traceability applies regardless of authorship.
4. `CLAUDE.md` is a controlled document (changes via PR).
5. AI may propose new SRS/SVS entries; they become applicable only after human
   approval and merge.
6. AI shall obtain human approval before adding, removing, or upgrading any
   third-party dependency (including build plugins), presenting license and
   distribution impact beforehand. All dependencies are recorded in the
   Software Reuse File (`docs/reuse-file.md`).
